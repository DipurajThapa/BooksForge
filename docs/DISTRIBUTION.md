## 3. Code-signing identities

| Platform | What we need | How to obtain | Stored in |
|----------|--------------|---------------|-----------|
| macOS | Apple Developer Program membership ($99/yr); Developer ID Application certificate | Enroll at <https://developer.apple.com/programs/>; create cert via Xcode or developer portal; export `.p12` | GitHub Secrets: `APPLE_DEVELOPER_ID_CERT` (base64-encoded `.p12`), `APPLE_DEVELOPER_ID_PASSWORD`. |
| macOS notarisation | Apple ID with notarisation rights, app-specific password, 10-char Team ID | Apple ID Settings â†’ Sign-In and Security â†’ App-Specific Passwords | Secrets: `APPLE_NOTARIZATION_APPLE_ID`, `APPLE_NOTARIZATION_PASSWORD`, `APPLE_NOTARIZATION_TEAM_ID`. |
| Windows | EV Code-Signing Certificate from a CA on Microsoft's accepted list (DigiCert, Sectigo, etc.); typically $300â€“500/yr; **MUST be EV** to avoid SmartScreen warnings | Buy from a CA, attest identity (D&B verification or notarised paperwork), install on a hardware token (or buy a cloud-HSM-backed cert) | Secrets: `WINDOWS_SIGNING_CERT` (base64 `.pfx`) and `WINDOWS_SIGNING_PASSWORD`. **For HSM-backed EV certs, the secret stores credentials for the HSM, not the cert itself.** |
| Tauri updater manifest | Ed25519 keypair generated by Tauri (`tauri signer generate`) | Local one-time generation; private key stays in CI, public key is committed in `tauri.conf.json` | Secrets: `TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`. Public key in `tauri.conf.json` (committed). |

**Key custody:** the macOS `.p12`, the Windows `.pfx` (or HSM
credentials), and the Tauri updater private key live **only** in
GitHub Secrets and on the founder's offline backup. They are never
checked in, never logged, never exported from the runner.

**Rotation policy:** rotate signing certs annually (with their
expiry); rotate the Tauri updater key on any suspected compromise
(this requires shipping a new app version with the new public key
embedded, so plan accordingly â€” ideally only the founder needs to
do this).

## 4. Notarisation flow (macOS)

After Tauri builds a signed `.app`:

1. Bundle into DMG (Tauri does this).
2. Submit DMG to Apple's notary service via `xcrun notarytool submit
   --wait`.
3. Apple returns an approval (or rejection with reasons).
4. Staple the approval to the DMG (`xcrun stapler staple`).
5. Verify (`spctl --assess --verbose`).

If notarisation fails, the build fails CI â€” we **never** publish a
non-notarised DMG to GitHub Releases.

`tauri-action` handles steps 2-4 automatically when
`APPLE_NOTARIZATION_*` secrets are set.

## 5. Auto-updater channel strategy

| Channel | Audience | `latest.json` path | Notes |
|---------|----------|-------------------|-------|
| `stable` | All users (default) | `latest.json` or `stable/latest.json` | Promoted manually after 1 week on `beta`. |
| `beta` | Opt-in via Settings | `beta/latest.json` | Released from tagged `v*-beta.*`. |
| `dev` | Internal | `dev/latest.json` | Each main-branch merge. Off-limits to public users. |

The user sees a single toggle in *Settings â†’ Updates*: ON / OFF. The
channel selector is hidden behind a *Show advanced* control. Default
channel is `stable`, default toggle is ON, with a clear "Skip and
remind me later" option on every prompt.

The updater runs **at most once per launch** and never blocks the
app from starting.

## 6. Versioning + release cadence

- **Pre-1.0:** `0.X.Y` SemVer. Tag `v0.X.Y` triggers `release.yml`.
- **MVP gate (v0.1.0):** all 9 acceptance criteria from
  `outputs/MVP_SCOPE.md Â§6`.
- **Public-beta gate (v0.5.0 target):** every CRITICAL + HIGH item in
  `EXTERNAL_AUDIT_BACKLOG.md` closed.
- **1.0 GA:** all open audit items closed; pricing + license + EULA
  finalised.

Release cadence pre-1.0: tag a `v0.X.Y-rc.N` candidate, soak for ~7
days on internal alpha, then promote to `stable`. Hotfixes (security
or data-loss only) skip the soak.

A `CHANGELOG.md` lives at `booksforge/CHANGELOG.md` and is generated
from Conventional Commits via `git-cliff`. The release workflow
attaches the relevant changelog slice to each GitHub Release.

## 8. Rollback strategy

If a published release ships a regression:

1. Within minutes: yank the GitHub Release (mark it as "draft" again
   or delete; the Tauri updater stops serving it).
2. Within hours: tag `v0.X.Y+1` from the previous good commit + the
   minimum hotfix; release.
3. Affected users: the next update check pulls them onto the hotfix.
   Users who already updated to the broken release see the hotfix on
   their next launch.

There is no "downgrade" automation in MVP. If a release destroys data
(should never happen, but plan for it), the recovery path is the
project bundle's snapshot system, not an installer downgrade.
